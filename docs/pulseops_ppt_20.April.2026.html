The Problem We Solve

Operations teams manage a complex ecosystem of systems — GCP, Kubernetes, Workloads, Dataflow jobs, PGSQL databases, Sailpoint IIQ, Forgerock IGA — with no unified automated monitoring available today:

❌ No Unified Monitoring

Operations must monitor GCP infrastructure, Kubernetes clusters, Workloads, Dataflow jobs, PGSQL databases, and identity systems separately. No single platform exists to consolidate all systems in one place.

❌ No Automated Incident Acknowledgment

Incidents sit in "New" status until manually acknowledged. There is no automated acknowledgment process today — leading to delayed response times and SLA breaches.

❌ No Automated SLA Reports

Client requires Uptime SLA Reports and Incident SLA Reports that are not available today. Monthly reports involve days of manual data gathering, calculation, and formatting.

❌ No Intelligent SLA Calculation

SLA calculations today do not account for business hours. Incidents resolved outside business hours inflate resolution times unfairly. There is no business-hour-aware SLA engine.

❌ No Uptime Monitoring with Downtime Exclusion

Application uptime is measured without considering planned maintenance. Change requests and ServiceNow maintenance windows are not factored in — resulting in inaccurate SLA numbers.

❌ No Proactive System Health Visibility

Application availability is measured reactively — only after users report outages. No automated minute-by-minute polling or historical uptime trend tracking exists.

⚠️ Without unified monitoring, operations teams spend significant manual effort switching between systems. Late incident response adds 30-40% to resolution time and risks SLA penalties.

Our Solution: PulseOps V3

A unified operations intelligence platform that addresses every gap with automation, intelligence, and extensibility:

✅ Unified Monitoring Dashboard

Single pane of glass for all systems — applications, services, incidents, SLA compliance. Real-time SSE updates. No more switching between tools.

✅ Automated Incident Acknowledgment

✅ Automated SLA Reports (Client Deliverables)

One-click Uptime SLA Report, Incident Resolution SLA Report, and Incident Response SLA Report. PDF export ready for client delivery.

✅ Intelligent Business-Hour SLA Calculation

SLA resolution and response times calculated using business hours only — weekends and off-hours excluded. Per-priority thresholds (P1: 4h, P2: 8h, P3: 24h, P4: 72h). Fair, accurate SLA metrics.

✅ Intelligent Uptime with Planned Downtime Exclusion

Uptime monitoring automatically checks ServiceNow change requests and maintenance windows. Planned downtime is excluded from SLA calculations — ensuring accurate, fair uptime numbers.

✅ Modular & Extensible — Future Ready

Add new monitoring modules (Sailpoint, Forgerock IGA, PGSQL, ElasticSearch) as plug-and-play extensions. No code changes to existing platform. Zero downtime deployment.

What Happens Behind the Scenes

Module Scanner

Scans dist-modules/ directory for available module bundles (manifest.js + constants.json + api/)

Database Registration

Module metadata is persisted to system_modules table. Schema auto-provisioned from Schema.json.

Dynamic API Mount

Express router loaded dynamically at /api/{moduleId}/*. JWT auth applied automatically.

UI Manifest Load

ES module bundle fetched via dynamic import() with cache-busting. Nav items appear instantly.

Lifecycle Hooks

onEnable() fires — starts background services (pollers, sync). onDisable() stops them cleanly.

Module Contract

Every module is self-contained and follows a strict contract:

FilePurpose
manifest.jsxUI entry — navItems, getViews(), getConfigTabs()
constants.jsonModule metadata (id, name, version, roles)
Schema.jsonDatabase table definitions (auto-provisioned)
api/index.jsExpress router + lifecycle hooks
config/*.jsonUI text, errors, messages, URLs

Developer Experience: A module template (_template/) provides scaffolding for new modules. Build, drop, enable — that's it. No core platform changes needed.

Engineering Standards & Quality

PulseOps V3 is built to enterprise-grade standards — every file, every endpoint, every component:

🛡️ Security

  • JWT + HTTP-only cookies — no token in localStorage
  • bcrypt password hashing — industry-standard security
  • RBAC — 5 roles with granular permissions
  • Rate limiting — per-endpoint throttling
  • Helmet security headers — XSS, CSRF protection
  • CORS configuration — origin whitelisting
  • Password redaction — credentials never exposed in API responses

🏗️ Code Quality

  • Zero hardcoded strings — all text in JSON config files
  • Comprehensive file headers — purpose, architecture, dependencies
  • Structured logging — Winston + DB persistence at all levels
  • Consistent error handling — standardized error format
  • API documentation — Swagger/OpenAPI integration
  • Module template — scaffolding for new modules
  • Reusable components — shared design system across modules

📊 Database Architecture

  • 9 core tables — users, roles, permissions, sessions, logs, modules, config
  • 6 HealthCheck tables — config, categories, applications, poll results, SLA, downtime
  • 5 ServiceNow tables — config, SLA, business hours, sync history, auto-ack log
  • JSONB flexibility — key-value stores for extensible configuration
  • Auto-provisioning — module tables created from Schema.json on enable

🌐 Scalability

  • Stateless API — no in-memory caches or session state
  • DB-only dedup — safe for multi-pod Kubernetes deployment
  • Connection pooling — PostgreSQL pg-pool for high concurrency
  • Server-side pagination — limit/offset for large datasets
  • Field projection — only requested columns fetched from SNOW API

GKE Pod Architecture

  • Stateless Pods: No in-memory state. All state in Cloud SQL. Safe for scaling and restarts.
  • Horizontal Pod Autoscaler: Scale pods based on CPU/memory. Each pod serves both UI and API.
  • PersistentVolume: dist-modules/ directory shared across all pods via GCP Persistent Disk.
  • DB-only dedup: Auto-ack poller uses DB locking — safe for multi-pod execution.
  • Session management: JWT sessions stored in PostgreSQL — pod-safe invalidation.

GCP Services Used

  • GKE: Managed Kubernetes cluster for pod orchestration and auto-scaling.
  • Cloud SQL: Managed PostgreSQL with HA, automated backups, and failover.
  • Cloud Load Balancer: HTTPS termination, SSL certificates, health checks.
  • Container Registry: Docker image storage for PulseOps builds.
  • Cloud Build / CI/CD: Automated build and deploy pipeline.

🔄 Zero-Downtime Module Updates: Build module → push to PV → enable via UI → all pods pick up the new module instantly. No rolling restart needed.
